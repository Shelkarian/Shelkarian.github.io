# Эффекты и состояния

## 1. Контракт MVP

Состояние — серверный эффект с идентификатором, источником, интенсивностью, оставшейся длительностью и правилом stacking. Все числа ниже являются стартовыми балансными значениями для прототипа.

Единица периодического урона проходит правила типов из [математики урона](damage_math.md), но броня не применяется повторно. Тик, приходящийся точно на момент окончания, не выполняется. Таймеры используют серверное время; пауза карты останавливает их.

Сила состояния обозначается целым `stack`. Если источник задаёт дробное значение, оно округляется вниз до применения. Значение меньше `1` не создаёт состояние.

## 2. Правила stacking

В MVP используются четыре политики:

| Политика | Повторное применение |
| :--- | :--- |
| `add` | прибавляет stack до cap и обновляет длительность до `max(остаток, новая)` |
| `refresh` | stack становится `max(текущий, новый)`, длительность становится полной новой длительностью |
| `extend` | stack становится `max(текущий, новый)`, длительности складываются до duration cap |
| `max` | stack и длительность независимо становятся максимумом текущего и нового значения |

Состояния одного ID объединяются независимо от источника. Последний источник хранится для боевого лога, но не создаёт отдельную копию. Снятие части stack не меняет оставшуюся длительность. При stack `0` состояние удаляется немедленно.

| `stunned` | отдельный спецэффект; не создаётся обычным уроном MVP | полный запрет управления и движения | `max`, `1 stack` | стартово `1.50 с` | окончание таймера; стимулятор снимает через `1 с` | Self/Others: иконка и явная анимация |
| `slowed` | отдельный спецэффект | скорость умножается на `0.75` | `refresh`, `1 stack` | `5 с` | окончание; очищающий стимулятор | Self: иконка/таймер; Others: без иконки, видно по движению |
| `fear` | отдельный психический источник | запрет добровольно приближаться к источнику ближе `3 тайлов`; прочие действия разрешены | `refresh`, `1 stack` | `6 с` | разрыв видимости на `3 с`; успокоительное мгновенно | Self: иконка и выделение источника; Others: нет |

## 4. Взаимодействия и приоритеты

1. Прямой урон рассчитывается и применяется.
2. Проверяются пороги состояний по итоговым компонентам урона.
3. Применяются новые stack и немедленные очищающие взаимодействия типа heat/cold.
4. Проверяется потеря дееспособности или смерть.
5. Периодические тики, запланированные на тот же timestamp, выполняются после прямого попадания только для пережившей его цели.

Множители одного типа от состояний перемножаются и затем ограничиваются диапазоном `0.5..2.0`. Например, `corroded 2` даёт `status_multiplier = 1.20`, а не два независимых множителя `1.10`.

`Stagger` и `stunned` могут существовать одновременно; действует более строгий запрет, а таймеры идут параллельно. `Slowed`, `chilled`, активный блок и истощение перемножают скорость, но итоговая управляемая скорость ограничена минимумом `30%`. Полный stun задаёт `0%` и обходит этот предел.

Смерть удаляет `stagger`, `stunned`, `slowed` и `fear`. `Bleeding`, `burning`, `chilled`, `poisoned`, `corroded` и причина asphyxiation остаются на теле до обычной очистки или удаления сущности.

## 5. Видимость и UI

- `Self` всегда видит название, целое число stack и визуально округлённую вверх оставшуюся длительность, кроме бессрочной фазы `asphyxiating`.
- `Others` получают только указанную в таблице информацию без точного таймера и, если не сказано иначе, без числа stack.
- Наблюдатели и администраторы могут видеть полные серверные данные в диагностическом режиме; это не игровая информация персонажа.
- Скрытое для Others состояние не должно создавать публичный alert через общий UI. Его заметные анимации появляются только на пороге из таблицы.
- Очистка подтверждается исчезновением alert и коротким сообщением действующему игроку. Неуспешная очистка не проигрывает эффект успеха.

## 6. Edge cases

- Повторное наложение на cap всё ещё обновляет длительность согласно политике, но не создаёт лишние stack.
- Два применения в один тик обрабатываются в стабильном порядке ID событий; cap проверяется после каждого.
- Очистка и тик в одно время: очистка выполняется первой, если её действие завершилось не позже timestamp тика.
- Источник удалён или умер: уже наложенное состояние продолжает действовать; лог сохраняет последний доступный ID источника.
- Цель имеет иммунитет к соответствующему типу (`R=0`): прямой урон равен нулю и состояние по порогу не накладывается. Явное наложение состоянием в обход урона обязано отдельно проверить иммунитет.
- Состояние с периодическим уроном не рекурсивно накладывает само себя и не запускает `on_hit` оружия.
- Heat и cold, пришедшие в одном смешанном пакете, сначала оба наносят урон; затем очистки взаимно уменьшают уже существующие/new stack по одному разу каждая.
- Переподключение игрока, смена карты сущностью и сохранение/загрузка не сбрасывают серверные состояния и таймеры.
- Недоступный способ лечения не удаляет состояние автоматически: каждое состояние всё равно заканчивается по таймеру либо имеет базовый доступный контр-инструмент в раунде.

## 7. Acceptance criteria

1. Каждое состояние из таблицы имеет серверные `stack`, cap, длительность, политику повторного применения и хотя бы один работающий способ очистки.
2. Автотесты покрывают все четыре stacking-политики, достижение cap, частичную очистку и очистку одновременно с тиком.
3. Пороги используют итоговый урон после блока, резистов и брони; parry и иммунитет не создают состояния.
4. DoT наносит ровно указанный урон и число тиков без дополнительного тика на границе окончания.
5. `Stagger` срывает wind-up, запрещает боевые действия на своей длительности и использует правило `max`, а не сложение.
6. Комбинация модификаторов движения никогда не опускает управляемую цель ниже `30%`, кроме полного stun.
7. UI владельца и наблюдателя соблюдает колонку «Видимость»; скрытые stack и таймеры не передаются через публичные alerts.
8. Состояния корректно переживают переподключение и не дублируются после сериализации.
9. Боевой лог фиксирует ID состояния, изменение stack, источник, причину очистки и каждый фактически применённый тик урона.
