# Математика урона

## 1. Назначение и границы

Этот документ задаёт единственный порядок расчёта прямого урона для MVP. Числа являются стартовыми балансными значениями для прототипа. Расчёт выполняется авторитетно на сервере и не использует случайность, критические удары или зоны тела.

Урон представлен пакетом именованных компонентов, например `{ slashing: 12, heat: 4 }`. Отрицательные компоненты запрещены; лечение является отдельной операцией.

## 2. Типы урона

| Тип | Группа | Типичные источники | Обычная защита |
| :--- | :--- | :--- | :--- |
| `slashing` | physical | меч, топор, стекло | порезостойкость брони |
| `piercing` | physical | копьё, пуля, игла | защита от прокола |
| `blunt` | physical | молот, кулак, падение | амортизация |
| `heat` | thermal | огонь, лазер, горячая среда | теплоизоляция |
| `cold` | thermal | криогеника, холодная среда | теплоизоляция |
| `poison` | biological | токсины, яд, заражённая пища | герметичность/биозащита или видовой резист |
| `acid` | chemical | кислота, едкие реагенты | химическая защита |
| `asphyxiation` | internal | нехватка кислорода, удушение | только специальный резист/иммунитет |

Огонь наносит `heat`, кислота — `acid`; они не объединяются в общий тип burn. `Poison` и `asphyxiation` не являются физическим уроном и не уменьшаются обычной бронёй.

## 3. Данные защиты

Для каждого типа у существа вычисляются:

- `R` — видовой/системный множитель сопротивления, от `0.0` до `2.0`; `0` означает иммунитет, `1` — без изменения, `2` — двойной урон;
- `A` — сумма плоского поглощения подходящих слоёв брони, не меньше `0`;
- `M` — произведение множителей подходящих слоёв брони, ограниченное диапазоном `0.10..1.50`.

Стартовые пресеты одного надетого комплекта:

| Броня | Защищаемые типы | `A` | `M` |
| :--- | :--- | ---: | ---: |
| Лёгкая | slashing, piercing | `2` | `0.85` |
| Бронежилет | piercing | `5` | `0.65` |
| Тяжёлая | slashing, piercing, blunt | `4` | `0.70` |
| Теплозащитная | heat, cold | `3` | `0.60` |
| Химзащита | acid, poison | `3` | `0.50` |

Обычная броня не задаёт `A/M` для `asphyxiation`. Герметичный костюм может защищать от внешнего `poison`, только если источник явно помечен как контактный или воздушный; введённый внутрь poison обходит этот слой. Источник обязан передать route (`contact`, `airborne`, `injected`, `internal`).

Если несколько предметов защищают один участок в MVP без зон тела, применяется один лучший плоский показатель `A = max(A_i)` и один лучший множитель `M = min(M_i)`. Это исключает чрезмерный рост защиты от многослойной одежды. Видовой `R` применяется отдельно.

## 4. Порядок расчёта

Для каждого положительного исходного компонента `D0[type]`:

1. **Источник:** `D1 = D0 × source_multiplier`. Стартово `source_multiplier = 1.0`; допустимый диапазон `0.25..3.0`.
2. **Защитное действие:** `D2 = D1 × defense_multiplier`. Обычный удар — `1.0`, блок задаётся в [ближнем бою](melee.md), парирование завершает весь расчёт нулём.
3. **Вид/системный резист:** `D3 = D2 × R`. При `R = 0` компонент помечается иммунным и остаётся нулём.
4. **Броня:** для неиммунного компонента `D4 = max(0, D3 - A) × M`.
5. **Уязвимость состояния:** `D5 = D4 × status_multiplier`. Стартово `1.0`, допустимый итоговый диапазон `0.5..2.0`.
6. Неокруглённые `D5` всех типов суммируются.
7. Итог округляется до ближайшего целого по правилу `0.5` вверх.

Формула компонента:

```text
D5[type] = max(0, D0[type] × source_multiplier × defense_multiplier × R[type] - A[type])
           × M[type] × status_multiplier[type]
```

### Минимальный урон

Если атака не парирована, содержит хотя бы один положительный неиммунный компонент и после шага 7 дала `0`, итог становится `1` урона. Это один пункт за весь пакет, а не за каждый тип. Тип минимального урона — неиммунный тип с наибольшим `D5`; при равенстве используется порядок типов из раздела 2.

Полный иммунитет всех компонентов и успешное парирование дают `0`. Значение `A`, полностью поглотившее слабый удар, всё равно допускает `1` chip damage; абсолютная защита должна выражаться через `R = 0`, а не бесконечную броню.

### Распределение округлённого результата

Для логов и порогов состояний итог по типам распределяется методом наибольших остатков: сначала берётся `floor(D5[type])`, затем недостающие единицы выдаются типам с наибольшей дробной частью. Равенство разрешается порядком из раздела 2. Сумма компонентов всегда равна итоговому урону.

## 5. Примеры

**Меч против тяжёлой брони:** `18 slashing`, `R=1`, `A=4`, `M=0.70`.

```text
(18 - 4) × 0.70 = 9.8 → 10 slashing
```

**Меч заблокирован щитом и попал в тяжёлую броню:**

```text
(18 × 0.35 - 4) × 0.70 = 1.61 → 2 slashing
```

**Смешанный пакет против теплозащиты:** `{ blunt: 6, heat: 8 }`; для blunt нет брони, для heat `A=3`, `M=0.60`.

```text
blunt = 6
heat = (8 - 3) × 0.60 = 3
total = 9
```

**Слабый poison против химзащиты:** `2 poison`, `A=3`, `M=0.50`, неиммунная цель.

```text
max(0, 2 - 3) × 0.50 = 0 → minimum 1 poison
```

**Удушение:** `5 asphyxiation`, обычная тяжёлая броня не подходит, `R=1`.

```text
5 × 1 = 5 asphyxiation
```

## 6. Броня и состояния

- Броня влияет только на прямой пакет источника. Периодический урон состояния повторно проходит броню, только если строка состояния явно указывает `armor each tick`; в MVP таких состояний нет.
- Порог применения состояния проверяется по распределённому итоговому урону соответствующего типа, уже после защиты.
- Успешное парирование не применяет состояния атаки. При `0` из-за полного иммунитета состояния этого типа также не применяются.
- Chip damage может запустить состояние только если отдельно выполнен его порог; одного факта попадания недостаточно.
- Блок не защищает от уже активных DoT-состояний.

## 7. Edge cases

- Пустой пакет, только нулевые компоненты или `source_multiplier = 0` дают `0` и не запускают on-hit эффекты.
- Неизвестный тип урона отклоняется валидацией прототипа; он не превращается в typeless damage.
- `NaN`, бесконечность и отрицательные значения отклоняются до изменения здоровья.
- Множители сначала перемножаются с полной точностью и только затем ограничиваются; промежуточное округление запрещено.
- При нескольких слоях брони используются только лучшие `A` и `M`, даже если они взяты с разных подходящих предметов.
- Снятие брони и смена состояния в тот же тик разрешаются по авторитетному порядку событий сервера до момента попадания.
- Overkill не переносится на другие цели. Здоровье может быть ограничено своей системой, но лог урона хранит рассчитанное значение и фактически применённое значение отдельно.
- Периодический урон рассчитывается отдельными пакетами на каждом тике и не объединяется ради обхода минимального урона.

## 8. Acceptance criteria

1. Реализация принимает пакет из всех восьми типов и возвращает детерминированные целочисленные компоненты, сумма которых равна итогу.
2. Порядок `source → defense → resistance → flat armor → armor multiplier → status multiplier → rounding` закреплён автоматическими тестами.
3. Тесты примеров из раздела 5 возвращают соответственно `10`, `2`, `9`, `1` и `5`.
4. Полный иммунитет и парирование дают `0`, а полностью поглощённый бронёй неиммунный пакет даёт ровно `1` за атаку.
5. `Asphyxiation` игнорирует обычную броню; heat, cold, poison и acid используют только явно подходящую защиту.
6. Многослойная броня выбирает лучшие `A/M`, не суммирует плоскую защиту и соблюдает предел множителя `0.10..1.50`.
7. Невалидные числа и неизвестные типы отклоняются без частичного применения урона.
8. Боевой лог может показать исходный пакет, защитное действие, `R/A/M`, итог по типам и причину нулевого результата.
