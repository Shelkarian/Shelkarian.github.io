# Правила публикации

Как добавлять и обновлять страницы этого диздока. Цель — чтобы документация оставалась
достоверной и собиралась без ошибок.

## 1. Происхождение и статус

Диздок описывает и *замысел*, и *текущую реализацию*. Их нельзя смешивать молча.

Слово `перенесено` означает только происхождение страницы: текст импортирован из старого документа.
Оно не означает, что механика согласована или реализована. Статус указывается отдельно.

Если отдельного маркера статуса нет, страница считается **черновиком**. Благодаря этому старые
страницы не нужно массово помечать, а статус `Принято` всегда является явным решением.

- Нереализованную механику помечаем явно: **`(не реализовано)`** и ссылку на issue.
  Нельзя описывать замысел так, будто он уже в игре.
- Статус страницы/раздела ставим первой строкой в виде маркера-цитаты:
  - `> **Принято** — правила согласованы и могут использоваться для постановки задач.`
  - `> **WIP** — раздел в работе, возможны изменения.`
  - `> **Черновик** — не согласовано.`
  - `> **Устарело** — не соответствует текущему контенту, см. …`
- Устаревшую страницу не удаляем молча: помечаем `Устарело` и заводим задачу на актуализацию.

## 2. Оглавление

- Страница считается опубликованной, только когда добавлена в `src/SUMMARY.md` —
  mdBook строит сайт строго по нему, файлы вне `SUMMARY.md` на не публикуются.
- Одна система = одна страница.

## 3. Связь с контентом

- Каждая страница про игровой контент **обязана** иметь секцию «Реализация» со ссылками на
  конкретные файлы.
- Ссылаемся на реальные файлы, не нужно пересказывать разработку построчно: текст описывает *замысел*.

## 4. Язык и стиль

- Язык — русский. Термины берём из игровой локали (`job-name-*`, `job-description-*`),
  без разнобоя; спорные термины фиксируем в глоссарии.
- Заголовки — обычным предложением, не Title Case.
- Длина строки желательно ≤ 120 символов (как в коде проекта).

## 5. Оформление

- Диаграммы связей — блоки ` ```mermaid ` (плагин подключён в CI).

## 6. Процесс (Git / ревью)

- Заметная правка → ветка → Pull Request → ревью. Мелкие текстовые правки можно вносить
  прямо в `main`.
- **Не мержим**, если сборка красная или есть битые ссылки (проверяется автоматически, см. §7).
- Перед PR прогоняем локально `mdbook serve` и глазами проверяем страницу.
- Один PR = одна логическая единица (одна система/департамент), а не свалка правок.

## 7. Сборка и деплой

- Push в `main` = публикация: workflow `.github/workflows/deploy.yml` сам собирает книгу
  и деплоит её на GitHub Pages. После мержа убеждаемся, что прогон зелёный и страница появилась.
- В сборку включена проверка ссылок **`mdbook-linkcheck`**: любая битая внутренняя ссылка
  роняет сборку, поэтому PR с битой ссылкой не пройдёт. Внешние (http) ссылки не проверяются,
  чтобы сборка не зависела от доступности сторонних сайтов.
- Версии инструментов (`mdbook` и плагины) меняем осознанно и отдельным PR.
